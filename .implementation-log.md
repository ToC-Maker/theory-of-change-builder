### PR 0 — Foundation

**Status**: complete on branch `figma/pr-0-foundation`. Preflight green (352 tests; 320 workerd + 32 jsdom after review-fix cleanup).

**Review-pass fixes applied (Wave 1 → 3)**:

- StrictMode regression in `useGraphMutation`: `isMountedRef.current` was flipped false in cleanup but never re-armed on (re-)mount. Under React 19 `<StrictMode>` every effect runs mount → cleanup → mount, leaving the ref permanently false and silently disabling every parent notify in `npm run dev`. Fix: `isMountedRef.current = true` at the top of the effect body. Regression test wraps the hook in `<StrictMode>` via a full `<render>` + harness (renderHook alone does NOT trigger React 19's double-invoke).
- Stale `dataRef` when parent calls the exposed `setData` (AI-edit / external-state-replace path): the raw React setter bypassed the ref, so the next `mutate(prev => ...)` derived from pre-replacement state and wiped the AI edit. Fix: wrap exposed `setData` with `setDataExternal` that applies the updater against `dataRef.current` and writes the ref before calling the raw setter.
- `handleDragStart` had `nodeRefs` (a `useState`) in its `useCallback` dep list, so its identity churned on every node mount/unmount, invalidating React.memo on every node (defeated Task 0.4's whole point). Fix: mirror `nodeRefs` writes into a `useRef`; `handleDragStart` reads via the ref and drops `nodeRefs` from its deps. Other readers (connection rendering path, ConnectionsComponent prop) still use the state.
- Memo regression test was non-falsifying: counted `updateNodeRef` calls inside a stable-dep useEffect that fired only on mount, so the test passed even when `memo(NodeComponentInner)` was removed. Replaced with a React.Profiler-based render-counter using the `actualDuration/baseDuration` ratio (bailed-out commits stay < 0.3 of baseDuration; real renders are ≥ 0.3). Verified falsifying by removing `memo()` and seeing the assertion fail.
- ErrorBoundary typed `error` as `Error` but React's contract is `unknown` (`throw 'string'` is legal). A non-Error throw would crash the boundary itself. Fix: state.error: `unknown`, narrow with `instanceof Error` before dereferencing `.name`/`.message`/`.stack`, wrap non-Error throws in a synthetic Error. Added a regression test for `throw 'string'`.
- Speculative-export cleanup in `useGraphLayout`: deleted `classifyRegion` (87 LoC + 10 tests), `scheduleRectRefresh`/`RefreshState` exports (kept the function as an internal helper inside the hook), `getSnapshot` result field, `_setForceRender` placeholder. Migrated three inline `getLocalPosition` copies (ConnectionsComponent ×2, TheoryOfChangeGraph ×1) to use the export. PR 4/5 will reintroduce `classifyRegion`/`getSnapshot` alongside their first real consumer (TDD).
- Dead `useGraphMutation.isInputFocused` surface: every production caller imported the utility directly; nobody used the hook-bound variant. Removed.
- Test gaps: added `mutateDebounced` 200ms idle-commit test (fake timers), `mutateDebounced` + unmount race test, `commit` clears the idle timer (no double-fire), and the StrictMode + non-Error-throw tests mentioned above.
- Important refinements: type baseProps in `NodeComponent.memo.test.tsx` as `Partial<ComponentProps<typeof NodeComponent>>`, simplified `setData(() => next)` → `setData(next)` in `writeLocal` (the `() => next` form had no behavioral benefit — `next` is a captured constant), added the canonical React warning string with parens placeholders as a search anchor in the `useGraphMutation` header.

**Deviations / autonomous calls (original PR 0)**:

- **ESLint `react/jsx-no-bind` rule scoped to NodeComponent**: skipped. The plan's optional ESLint rule needs `eslint-plugin-react`, which is not installed (only `eslint-plugin-react-hooks` is); pulling it in just for this rule is disproportionate. Coverage shifted to the `useCallback` audit in `TheoryOfChangeGraph.tsx` (toggleHighlight + handleDragStart + handleDragEnd) plus the `NodeComponent.memo.test.tsx` regression test, which fails if a future contributor re-introduces inline callbacks.

- **`mutateDebounced` foot-gun semantics**: the plan says "drops the earlier write silently" for same-key-different-properties; the actual implementation keeps both writes in live state (writeLocal applies both synchronously) but merges the parent NOTIFY into one. The "silent drop" is in the notify cadence, not the value emitted. Documented in the file header and pinned by the test "emits a SINGLE parent notify when two same-key writes target different properties (foot-gun)".

- **`@testing-library/jest-dom/vitest` setupFiles**: does not auto-register `cleanup()` between RTL tests. Each RTL test file explicitly calls `cleanup()` in `afterEach`. (Found while debugging a DOM-bleed failure in NodeComponent.memo.test.tsx.)

- **Layout-setting sliders in EditToolbar (curvature, textSize, paddings)**: rely on `useGraphMutation`'s built-in 200ms idle timer rather than an explicit `commit` on pointerup. The slider control is inside EditToolbar and doesn't expose pointer events through to TheoryOfChangeGraph; PR 1 deletes EditToolbar entirely, so this is bridge-only. The L3 fix still holds (one notify per gesture).

- **`fontFamily` picker**: kept as discrete `mutate` (one selection event, not streaming), so the change reaches localStorage immediately rather than after a 200ms idle.

- **`isContentEditable` in jsdom**: returns `undefined` for elements where `contentEditable="true"` was set as an IDL property (instead of attribute). `isInputFocused` checks three forms (`isContentEditable`, `contentEditable === 'true'`, and the lowercase attribute) so real browsers and jsdom both match.

- **Smoke test removed**: `tests/frontend/_smoke.test.tsx` was kept just long enough to confirm the jsdom project booted, then deleted in the same commit as the first real hook test landed (Task 0.1).

### PR 1 — TopBar + EditToolbar deletion

**Deviations / autonomous calls**:

- **Tasks 1.1 + 1.2 merged into one commit**. The TopBar shell imports FileMenu/FormatMenu/HelpPanel/MobileMenu; splitting requires throwaway stubs in 1.1 that 1.2 immediately replaces. One commit keeps the diff coherent. Commit subject still names task 1.1 (the umbrella task); 1.2 is mentioned in the body.
- **`onShareClick` bridge via CustomEvent**. The new TopBar's Share button dispatches `window.dispatchEvent(new CustomEvent('toc:open-share'))`; the (still-mounted) `EditToolbar` listens and opens its existing share dropdown. Short-lived shim that ends when Task 1.7 moves the dialog into `<EditToolbarRemnant>` and we pass a direct handler.
- **`isOwner` lifted to `App.tsx`**. Required so `FileMenu` can owner-gate the Delete item without prop-drilling through `TheoryOfChangeGraph`. Reading the value from the `getChartByEditToken` response in App's `loadData` and from the polling loop (still in EditToolbar) is duplicate; the duplicate goes away when EditToolbar is deleted in Task 1.7.
- **`handleDeleteChart` migrated to `App.tsx`** ahead of plan (originally in Task 1.2). Same reason as `isOwner`: TopBar mounts at App level and needs the handler. The duplicate copy stays in EditToolbar until Task 1.7.
- **`saveError` plumbing**. Plan §0 red-team finding "SaveIndicator failure-path display" calls for a distinct error state with `reportError` ping; implemented as `useState<SaveError>` in App, threaded into TopBar, and updated from all four `ChartService.updateChart(...)`-catch sites (manual edit, JSON upload, undo, redo). `SaveIndicator` itself debounces the report via a fingerprint ref so re-renders don't spam.
- **`HelpPanel` "Replay tutorial"** is implemented as `localStorage.removeItem('graph-tutorial-seen')` + `window.location.reload()`. The `<GraphTutorial>` component already gates on that flag, so this is the smallest possible link; no prop chain.
- **`src/data/keyboardShortcuts.ts` is data-only**. The existing `src/utils/keyboardShortcuts.ts` (platform-aware helpers `isMac()`, `shortcuts.undoDisplay()`) stays as-is; the new data file imports `isMac()` from it and exposes a structured `keyboardShortcutGroups` array. `useKeyboardShortcuts.ts` still keeps its hard-coded key checks for now (parity with PR 0); the data file is the canonical source for HelpPanel rendering. Future PRs may unify.
