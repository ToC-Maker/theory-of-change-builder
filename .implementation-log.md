- The main context only orchestrates; all code edits happen in sub-agents.

### PR 0 — Foundation

**Status**: complete on branch `figma/pr-0-foundation`. Preflight green (352 tests; 320 workerd + 32 jsdom after review-fix cleanup).

**Review-pass fixes applied (Wave 1 → 3)**:

- StrictMode regression in `useGraphMutation`: `isMountedRef.current` was flipped false in cleanup but never re-armed on (re-)mount. Under React 19 `<StrictMode>` every effect runs mount → cleanup → mount, leaving the ref permanently false and silently disabling every parent notify in `npm run dev`. Fix: `isMountedRef.current = true` at the top of the effect body. Regression test wraps the hook in `<StrictMode>` via a full `<render>` + harness (renderHook alone does NOT trigger React 19's double-invoke).
- Stale `dataRef` when parent calls the exposed `setData` (AI-edit / external-state-replace path): the raw React setter bypassed the ref, so the next `mutate(prev => ...)` derived from pre-replacement state and wiped the AI edit. Fix: wrap exposed `setData` with `setDataExternal` that applies the updater against `dataRef.current` and writes the ref before calling the raw setter.
- `handleDragStart` had `nodeRefs` (a `useState`) in its `useCallback` dep list, so its identity churned on every node mount/unmount, invalidating React.memo on every node (defeated Task 0.4's whole point). Fix: mirror `nodeRefs` writes into a `useRef`; `handleDragStart` reads via the ref and drops `nodeRefs` from its deps. Other readers (connection rendering path, ConnectionsComponent prop) still use the state.
- Memo regression test was non-falsifying: counted `updateNodeRef` calls inside a stable-dep useEffect that fired only on mount, so the test passed even when `memo(NodeComponentInner)` was removed. Replaced with a React.Profiler-based render-counter using the `actualDuration/baseDuration` ratio (bailed-out commits stay < 0.3 of baseDuration; real renders are ≥ 0.3). Verified falsifying by removing `memo()` and seeing the assertion fail.
- ErrorBoundary typed `error` as `Error` but React's contract is `unknown` (`throw 'string'` is legal). A non-Error throw would crash the boundary itself. Fix: state.error: `unknown`, narrow with `instanceof Error` before dereferencing `.name`/`.message`/`.stack`, wrap non-Error throws in a synthetic Error. Added a regression test for `throw 'string'`.
- Speculative-export cleanup in `useGraphLayout`: deleted `classifyRegion` (87 LoC + 10 tests), `scheduleRectRefresh`/`RefreshState` exports (kept the function as an internal helper inside the hook), `getSnapshot` result field, `_setForceRender` placeholder. Migrated three inline `getLocalPosition` copies (ConnectionsComponent ×2, TheoryOfChangeGraph ×1) to use the export. PR 4/5 will reintroduce `classifyRegion`/`getSnapshot` alongside their first real consumer (TDD).
- Dead `useGraphMutation.isInputFocused` surface: every production caller imported the utility directly; nobody used the hook-bound variant. Removed.
- Test gaps: added `mutateDebounced` 200ms idle-commit test (fake timers), `mutateDebounced` + unmount race test, `commit` clears the idle timer (no double-fire), and the StrictMode + non-Error-throw tests mentioned above.
- Important refinements: type baseProps in `NodeComponent.memo.test.tsx` as `Partial<ComponentProps<typeof NodeComponent>>`, simplified `setData(() => next)` → `setData(next)` in `writeLocal` (the `() => next` form had no behavioral benefit — `next` is a captured constant), added the canonical React warning string with parens placeholders as a search anchor in the `useGraphMutation` header.

**Deviations / autonomous calls (original PR 0)**:

- **ESLint `react/jsx-no-bind` rule scoped to NodeComponent**: skipped. The plan's optional ESLint rule needs `eslint-plugin-react`, which is not installed (only `eslint-plugin-react-hooks` is); pulling it in just for this rule is disproportionate. Coverage shifted to the `useCallback` audit in `TheoryOfChangeGraph.tsx` (toggleHighlight + handleDragStart + handleDragEnd) plus the `NodeComponent.memo.test.tsx` regression test, which fails if a future contributor re-introduces inline callbacks.

- **`mutateDebounced` foot-gun semantics**: the plan says "drops the earlier write silently" for same-key-different-properties; the actual implementation keeps both writes in live state (writeLocal applies both synchronously) but merges the parent NOTIFY into one. The "silent drop" is in the notify cadence, not the value emitted. Documented in the file header and pinned by the test "emits a SINGLE parent notify when two same-key writes target different properties (foot-gun)".

- **`@testing-library/jest-dom/vitest` setupFiles**: does not auto-register `cleanup()` between RTL tests. Each RTL test file explicitly calls `cleanup()` in `afterEach`. (Found while debugging a DOM-bleed failure in NodeComponent.memo.test.tsx.)

- **Layout-setting sliders in EditToolbar (curvature, textSize, paddings)**: rely on `useGraphMutation`'s built-in 200ms idle timer rather than an explicit `commit` on pointerup. The slider control is inside EditToolbar and doesn't expose pointer events through to TheoryOfChangeGraph; PR 1 deletes EditToolbar entirely, so this is bridge-only. The L3 fix still holds (one notify per gesture).

- **`fontFamily` picker**: kept as discrete `mutate` (one selection event, not streaming), so the change reaches localStorage immediately rather than after a 200ms idle.

- **`isContentEditable` in jsdom**: returns `undefined` for elements where `contentEditable="true"` was set as an IDL property (instead of attribute). `isInputFocused` checks three forms (`isContentEditable`, `contentEditable === 'true'`, and the lowercase attribute) so real browsers and jsdom both match.

- **Smoke test removed**: `tests/frontend/_smoke.test.tsx` was kept just long enough to confirm the jsdom project booted, then deleted in the same commit as the first real hook test landed (Task 0.1).

### PR 3 — NodeEditor + EdgeEditor + MDXEditor v4

**Status**: complete on branch `figma/pr-3-editors`. Preflight green (45 frontend test files; 448 tests; +31 new this PR).

**Deviations / autonomous calls**:

- **Sub-commit 3b (MDXEditor v3 → v4)**: v4 dropped its transitive dependency on `@types/node` that was previously satisfying our use of `NodeJS.Timeout` in 7 sites (4 in `src/App.tsx`, 1 each in `ConnectionsComponent.tsx`, `GraphTutorial.tsx`, `useZoomPan.ts`). Replaced with `ReturnType<typeof setTimeout>` (canonical browser-portable form). No `@types/node` was added — purely a type-annotation migration. All 13 currently-imported MDXEditor symbols (`MDXEditor`, `headingsPlugin`, `listsPlugin`, `quotePlugin`, `thematicBreakPlugin`, `markdownShortcutPlugin`, `linkPlugin`, `toolbarPlugin`, `UndoRedo`, `BoldItalicUnderlineToggles`, `ListsToggle`, `BlockTypeSelect`, `Separator`) still resolve in v4 unchanged.
- **No `node-drag-start` CustomEvent indirection exists today**. Plan §3.2 anticipated a `node-drag-start` CustomEvent dispatch in `NodeComponent`/`TheoryOfChangeGraph` to be replaced with `onDragStartedElsewhere` callback. Grep confirms no such event exists in the current code — only `react-dnd`-style HTML5 `onDragStart` on the node `<div>`. PR 3 therefore designs `NodeEditor.onDragStartedElsewhere` proactively for PR 4's `usePointerDrag` to wire in (no callback exists to _replace_ yet; just _declare_ the prop shape).
- **Selection-key pattern for `useNodeProperties`**. Plan §3.1 keying convention is `'title-${nodeId}' / 'details-${nodeId}'`. For multi-selection, those keys would multiply (one per selected node, each with its own idle timer), and any commit would have to enumerate them. Implementation uses a single stable `selectionKey` (sorted ids joined with `|`), so `'title-{id}'` collapses to one key per selection-set. Trade-off: switching from {a,b} to {a,b,c} changes the key, so any unflushed buffer for {a,b} gets stranded (its live state has already been written via `mutateDebounced`, so the data is fine — only the per-key idle timer would never auto-fire for the orphaned key). Acceptable because the same selection-changed path also resets the local input buffer.
- **DetailsEditor "collapsed-with-content" preview**: plan §3.2 specifies "non-interactive markdown viewer placeholder while chunk loads". Implementation renders a tiny text-only preview of the raw markdown (`whitespace-pre-wrap line-clamp-3`) when collapsed-with-content, NOT a rendered markdown viewer. Rationale: rendering markdown requires loading MDXEditor's lexical/mdast machinery, which is exactly what we're trying to avoid. A future PR can swap in a tiny offline markdown renderer (e.g. `marked` at ~50 KB) if a richer preview is useful; today the raw markdown serves as a sufficient "click here to see/edit" hint.
- **NodeEditor anchor is the FIRST sorted selected id**, not a centroid. Plan §3.2: "anchored beside selected node". With multi-select, "beside _the_ node" is ambiguous; using the first sorted id makes the anchor stable across additions to the selection (Cmd+click extending the set doesn't make the editor jump to a different node). Centroid-anchored would be more visually obvious for multi-select but adds complexity; deferred unless user feedback surfaces it.
- **`useKeyboardShortcuts` lost the `setNodePopup` arg**. Previously the Esc/clear-selection handler closed the NodePopup modal as part of its cleanup. The anchored NodeEditor watches `highlightedNodes`, so clearing the selection auto-dismisses the editor — no extra plumbing needed.
- **`GraphTutorial` simplified from three steps to two**. The middle step "Click the details button" was the entry into the NodePopup modal; with the pencil icon gone, single-click on a node both selects AND opens the NodeEditor, so the step disappears. New flow: "Click a node to see its connections and edit it" → "Click to see connection details". `isModalOpen` state and the 100ms position-tracking interval (specific to the info-button hover step) also removed.
- **`viewportOffset` prop removed from `<ToC>`**. The prop was only consumed by NodePopup / EdgePopup modal-sizing math (scale + viewport-offset positioning). The anchored editors use `useAnchorPosition` (viewport-coords from `getBoundingClientRect`) directly, so the prop has nothing to feed. Propagated up: App.tsx no longer passes it.
- **`onDeleteConnection` plumbing removed from `ConnectionsComponent`**. Previously fed the EdgePopup's delete button. The anchored EdgeEditor calls its own `useEdgeProperties.deleteConnection` (atomic write through the mutate triad), so the parent doesn't need to plumb a separate callback. `disconnectSelectedNodes` in TheoryOfChangeGraph still owns the 2-node-selected disconnect button.
- **Test cleanup pattern**: NodeEditor.test.tsx and EdgeEditor.test.tsx install `afterEach(cleanup)` + `document.querySelectorAll('#anchor').forEach((el) => el.remove())`. RTL doesn't auto-cleanup between tests in this project (verified by tracing existing tests — `GeneralAccessSelector.test.tsx` and `LinkCopyRow.test.tsx` install the same `cleanup()` pattern). Anchor divs are appended to `document.body` inside the render helper, so we tear them down explicitly to avoid cross-test pollution.

**Status**: complete on branch `figma/pr-4-pointer`. Preflight green (52 frontend test files; 467 total tests; +19 new this PR — 14 `usePointerDrag` + 5 `NodeComponent.pointer`).

**Commit shape (3 commits instead of the orchestrator-suggested 4-5)**:

The orchestrator brief acknowledged 4.2 + 4.4 share a commit; the practical constraint of typecheck-coherence pulled 4.3 in too.
3. `feat(canvas): drop-preview ghost render at dragOverLocation` — Task 4.5 (independent addition).

The orchestrator's suggested final `docs(figma)` commit folds into this log + the commit message of commit 3 (the revert kit blurb is in the worker log).

**Deviations / autonomous calls**:

- **Module-scope `isCanvasGestureActive` ref lives in a sibling module** (`src/hooks/_canvasGestureState.ts`), not inside `useGraphLayout.ts`. The plan said "in `useGraphLayout` module scope or a sibling"; the sibling option keeps `useGraphLayout` focused on geometry and avoids a circular dep when PR 5 / PR 7 import the flag (those hooks also import `LayoutSnapshot` from `useGraphLayout`). The sibling exports `isCanvasGestureActive()`, `setCanvasGestureActive(v)`, and a `_resetCanvasGestureStateForTest()` helper that test files use in `afterEach` to avoid cross-test bleed.

- **`bindNode` per-id handler cache**. `React.memo(NodeComponent)` requires stable callback props (the PR 0 regression test pins this). My first cut returned a fresh `{ onPointerDown }` per `bindNode(id)` call, which made every parent render invalidate the memo. Fix: the hook caches `{ onPointerDown: (e) => startDrag(id, e) }` per node id in a `Map`; the per-id object identity is stable across re-renders. Live props/callbacks (`editMode`, `getSnapshot`, `onDragStart`) are read at call-time via refs, so the handler doesn't need to capture them.

- **Stale-node check uses a dataRef mirror**, not a React state-setter callback trick. My first cut tried `setDragState((prev) => { captured = prev; return prev; })` to read the latest state synchronously inside the document-level `pointerup` handler; this works on React 18 in `act()` but isn't guaranteed in concurrent-rendering modes (the state setter's updater may be called asynchronously). Fixed: the hook maintains a `dragStateRef` mirror that the document-level handler reads directly. Same for `data` (so the stale-node guard sees the freshest snapshot).

- **`onDrop` callback shape**: the hook calls `onDrop(target, draggedNodeId)`; `handleDrop` in the parent needs `pointerOffsetY` too (for the same-cursor-offset math). The parent stores the live `handleDrop` in a ref and reads `dragState?.pointerOffset.y` at the moment of `onDrop` invocation. Cleaner than threading a `pointerOffset` parameter through `usePointerDrag.onDrop` (the hook stays generic; the offset is hook-internal state).

- **`new-section` Region mapping**. The plan calls out four Region kinds (`node-slot`, `over-node`, `new-column`, `new-section`); the existing `handleDrop` only handles three (no path for "drag node into the gap between sections to create a new section"). I expose `isNewSection` on the `DragOverLocation` shape so PR 5+ can consume it without re-plumbing, but the current `handleDrop` ignores `isNewSection` for now — matches today's behavior. Marked as a follow-up.

- **`registerOnDragStartedElsewhere` plumbing**. PR 3 declared the NodeEditor prop shape but had no caller; PR 4 wires it via `nodeEditorDragStartRef` in `TheoryOfChangeGraph` (the ref is set by `NodeEditor`'s `useEffect` and read by the hook's `onDragStart` callback). The contract is one-shot: PR 3 documented "re-registers every render of the editor"; this PR's bridge writes to the same ref on every re-registration, so stale callbacks are naturally replaced.

- **Polling pause on App.tsx**. Wired via a `onDragActiveChange` callback prop on `ToC` rather than a context or window event. App holds an `isDragInFlightRef`; both polling effects in `ToCViewerOnly` and `ToCViewer` check it inside `syncData`. View-only mode still gets the ref (dormant — drag never enters), so the polling-effect code is identical between the two functions.

- **Drop-preview ghost is intentionally generic**. A translucent indigo rounded rectangle, not a clone of the dragged node. The plan's §4.5 acceptance says "render translucent post-drop node ghost at `dragOverLocation`" + "floating drag-cursor ghost follows the cursor"; I made these two visually identical (a single ghost following the cursor) because the `dragOverLocation` ghost was load-bearing only for the connection-handle silhouette in PR 5 (per the orchestrator brief: "keep the ghost render generic"). PR 5 can swap in a positioned drop-target indicator (e.g. a translucent column highlight) without re-architecting this render slot.

- **Touch test matrix**: jsdom can't execute real touch gestures, and no rodney binary is present on this worktree. The 10-item matrix from §4.6 is documented in the worker log (`.workspace/workers/pr-4.md`) with each item labeled "unit-test-covered" / "requires manual device". The unit-test-covered items are exercised by `usePointerDrag.test.ts`; the device-only items are gated on Chromium DevTools touch emulation or a real iPad. Per the orchestrator's escape hatch ("If you can't easily get the touch matrix run in this environment, mark items as 'unit-test-covered' / 'requires manual device' — don't block the PR on real-device tests"), the PR is not gated on real-device execution.

- **`handleDrop` source-node lookup moved inside the updater**. The old `handleDrop` used `findNodeLocation(draggedNode.id)` outside the updater, then referenced `draggedNode` (the full Node from a closure) inside. The new version still calls `findNodeLocation` outside (for the same-column fast path), but the cross-column path now does a fresh node-by-id walk inside the `setDataAndNotify` updater so it doesn't leak a stale `draggedNode` snapshot. Matches the "fresh ref of data inside updater" pattern in the hook.

- **No `excludeFromPan` regression test**. The selector swap is a 1-LoC change; covered transitively by manual integration testing. Adding a dedicated regression test would require mounting a panning context + a fake node — disproportionate. Marked as a deferred follow-up if pan/drag interactions regress.

**Tracked deviations / decisions:**

- **`<ConfirmModal>` primitive built from scratch (not pre-existing)**. The orchestrator brief flagged that the FileMenu Delete chart modal might or might not exist as a reusable primitive in PR 1. Inspection of PR 1's `src/components/top-bar/FileMenu.tsx:117` confirmed it used `window.confirm()` inline — no React modal existed. Per the escape-hatch instruction, I built `src/components/ConfirmModal.tsx` and retrofitted FileMenu to use it (replacing `window.confirm`). The retrofit also closes the red-team L4 "confirm() blocks event loop" Important finding (`plans/figma-redesign.md:200`), which was tracked for PR 5 anyway. Existing `FileMenu.test.tsx` was updated to interact with the modal's test-ids instead of mocking `window.confirm`.

- **`group` class for hover-reveal vs JS hover tracking**. The hover-× delete button (`ColumnDeleteAffordance`) is hidden by default and revealed via Tailwind's `:group-hover` selector. The surrounding column / section container carries `group`, the button carries `opacity-0 group-hover:opacity-100`. Pure CSS, no JS hover state, no global pointermove subscription. Same pattern for the gutter labels.

- **`ConnectionHandles` visibility uses React state, not CSS hover**. The handle dots' visibility is gated by NodeComponent's `isHovered || isHighlighted` React state (already tracked by NodeComponent for visual selection feedback). This avoids inconsistency between the React-tracked node hover state and a CSS-only handle visibility (which would diverge when, say, the node is mid-drag and the cursor leaves but the visual selection persists).

- **Path math factored to `connectionPath.ts` for shared use**. The select-2-then-button preview ghost (in `ConnectionsComponent.tsx:670-714`) and the new drag-to-connect in-flight ghost (in `TheoryOfChangeGraph.tsx`) compute the exact same cubic-bezier path between two points. Rather than duplicate the math, I extracted `buildConnectionPath(args)` into `src/components/canvas/connectionPath.ts`. Both callsites use it; a unit-test file (`connectionPath.test.ts`) pins the exact SVG `d` shape so regressions surface loudly.

- **`useConnectionDrag` duplicates `usePointerDrag`'s pointer-capture boilerplate**. The shapes are similar but not identical (different hit-test logic — DOM `elementFromPoint` vs layout snapshot, different drop payload — single nodeId vs DragOverLocation). Extracting a shared "useGesture" base would amount to ~30 lines of pointer-capture boilerplate. The cost of duplication is low; the risk of premature abstraction is high (PR 7's waypoint drag will be the third user, by which point the right shape will be clearer). Documented this rationale in `useConnectionDrag.ts` header comment.

- **Stale-node guard distinguishes source vs target**. `useConnectionDrag` checks both source and target node existence before firing `onConnect`. If source is gone: `error_name: 'stale-connection-source'`. If target is gone: `'stale-connection-target'`. Each path is its own `reportError` call, so cross-tab race classification stays clean.

- **`addConnection` is idempotent**. Re-dropping a handle on a node that's already connected does NOT add a duplicate edge and does NOT log to undo. The `areNodesConnected` short-circuit at the top of the callback handles this. The rationale is anti-spam for accidental re-drops; the `disconnectSelectedNodes` path remains the way to remove edges.

- **Section delete handles non-empty sections too**. Per the plan Task 5.3 acceptance: "for section delete: only allow when section has no columns with nodes, OR show confirm with node count". I went with the second option (always allow with count-aware confirm) since blocking the affordance entirely would leave non-empty sections un-deletable without first emptying every node — bad UX. The confirm body shows the total node count across all columns in the section.

- **Empty column body gets `cursor-cell` not `cursor-crosshair`**. Plan suggested `cell` OR `crosshair`. I chose `cursor-cell` because it has a thinner visual footprint (closer to a text-insertion-point shape) and matches the spirit of "click here to insert a node" better than crosshair (which feels more like a draw-tool affordance).

- **Connection-drag ghost is a dashed indigo path** (not the same look as the select-2 solid ghost). The select-2 ghost uses `getConfidenceStrokeStyle(75)` (the same style a real connection would take). For the in-flight drag ghost, I went with a dashed style (`strokeDasharray: '6 4'`) so it visually distinguishes "preview pending drop" from "rendered final connection" — useful when the user is mid-drag with the cursor over a valid target.

- **`baseData` mutation in `setDataAndNotify` is a pre-existing bug**. The PR 5 gutter test (`TheoryOfChangeGraph.gutter.test.tsx`) initially failed across test cases because `setDataAndNotify` updaters in TheoryOfChangeGraph mutate `prevData.sections` via `.splice()` (the surrounding `{ ...prevData }` is shallow). This leaks across tests sharing a module-level fixture. Fixing the underlying mutation is out of PR 5 scope; the test uses a `makeBaseData()` factory to deep-clone per-test. Logged as a follow-up for a future PR.

- **Existing `connectSelectedNodes` path preserved**. The select-2-then-button gesture still works alongside the new drag-to-connect. Both go through the same `useGraphMutation.mutate` write path. Plan Task 5.2 explicitly says "Existing select-2-then-button gesture preserved."
