# Legitimate Interest Assessment (LIA)

**Data Controller:** Theory of Change Tool

**Assessment Date:** 2026-04-23

**Data Processing Activity:** Collection of AI improvement data (session metadata, chat messages, graph snapshots) and cost-accounting data (per-user cumulative AI spend, anti-abuse anonymous identifier)

---

## **0\. Scope note: what this LIA does and does not cover**

This assessment covers two activities we rely on under Art. 6(1)(f) legitimate interests:

1. **Server-side AI-improvement logging** (primary activity: messages, snapshots, error reports).
2. **Anti-abuse cost enforcement** (secondary activity: cumulative-spend counters per user/visitor, plus three first-party cookies:
   - `tocb_actor_id`: a random UUID generated once per browser that binds the spend counter to that browser for up to 1 year,
   - `tocb_auth_link`: a server-HMAC-signed cookie (1-year Max-Age) that records which Auth0 user ID this browser has been signed in as, so cap attribution survives sign-out,
   - `tocb_anon`: a short-lived (24-hour) Turnstile session cookie that gates access to AI features for anonymous visitors).

**File uploads (Chat and Generate modes).** These are processed under a different legal basis and do not require an LIA. When a user attaches a file for AI analysis, the processing of that file by Anthropic to generate the requested response is performed under **Art. 6(1)(b)** (necessary for the performance of a contract, namely the user's explicit request for AI assistance). This LIA does not extend to that activity. The user can remove their files at any time by clearing the chat, deleting the chart, or submitting a data erasure request; the privacy policy discloses the flow.

**Bring-Your-Own-Key (BYOK).** Also outside this LIA: the encrypted storage of a user-supplied Anthropic API key and its per-request use exist only because the user explicitly asked to swap their own billing account in. Legal basis: Art. 6(1)(b) (contract performance for the user-initiated BYOK feature).

The remainder of this document addresses the Art. 6(1)(f) activities above.

## **1\. Purpose**

### 1A. AI improvement logging

We collect usage data from the AI-assisted Theory of Change builder to:

- **Evaluate AI quality:** Measure how well the AI assistant understands user requests and produces useful graph edits.
- **Identify failure patterns:** Detect cases where the AI misinterprets instructions, produces broken edits, or fails to handle certain graph structures.
- **Improve prompt engineering:** Use real interaction patterns to refine the system prompts and edit instruction format, leading to better outcomes for all users.

Without this data, we cannot systematically evaluate or improve the AI features that are central to the application's value.

### 1B. Anti-abuse cost enforcement

We offer AI features at no cost up to a per-person lifetime spending cap (currently $5 of upstream Anthropic usage). To prevent one anonymous visitor from repeatedly resetting that cap by refreshing or changing browsers, and to protect the Service's overall AI-spend budget, we:

- Store a cumulative per-person spend counter in `user_api_usage` (signed-in users keyed by Auth0 ID; anonymous visitors keyed by the per-browser UUID described below).
- Mint a random UUID on a visitor's first request and pin it in an HttpOnly first-party cookie (`tocb_actor_id`, 1-year lifetime) so the same browser's spend is tracked across successive requests. The UUID is generated by the server's cryptographic random source; it is not derived from the IP address or any other personal data. Each browser gets its own UUID, so two people sharing a NAT or CGNAT get independent caps.
- Set an HMAC-signed `tocb_auth_link` cookie (1-year Max-Age) on successful authentication. When that browser later makes anonymous requests (after sign-out), we consult the cookie and keep attributing spend to the authenticated user's cap row, closing the "log out to reset the cap" loophole. The cookie is a server-signed payload `{sub, exp}`; it is consulted only for cap attribution and never substituted for a JWT elsewhere.
- Require a Cloudflare Turnstile challenge for anonymous visitors, renewed at most every 24 hours via a short-lived `tocb_anon` cookie. This cookie only proves the challenge was solved; it does not hold any spend identifier.

Without these controls, any anonymous user could exhaust the shared AI budget trivially by refreshing the page, clearing cookies in a loop, or logging in and out.

## **2\. Necessity Test**

### 2A. AI improvement logging

**Is this processing necessary for the stated purpose?**

Yes. The data collected is the minimum needed to evaluate and improve AI performance:

- **Session metadata** (session ID, chart ID, user ID if logged in, user agent, timestamps): Required to group interactions into coherent sessions for analysis. Users are identified by Auth0 user IDs (opaque identifiers), not by email addresses or names.
- **Chat messages** (user prompts and AI responses): Required to understand what users asked for and how the AI responded. This is the primary signal for evaluating AI quality. Chat prompts are retained server-side in the logging tables. They are also cached client-side in the user's browser localStorage for session continuity; users can clear this via browser settings or the "Clear Chat" button.
- **Graph snapshots** (graph state after edits): Required to verify whether AI edits were correct and to identify structural patterns the AI handles poorly. The graph data itself is similar to what is stored in the charts table, but the logging captures the temporal sequence of edits linked to specific AI interactions.
- **Error / diagnostic reports** (type, message, HTTP status, truncated stack trace, request metadata, worker hostname): Required to surface silent failures and cap-enforcement edge cases that would otherwise be invisible on branch-preview deployments. Metadata never includes message content.

**Why alternatives are insufficient:**

- **Synthetic test data** cannot replicate the diversity of real user requests, graph structures, and phrasing patterns. Edge cases discovered in production are, by definition, not anticipated in synthetic tests.
- **Aggregated metrics alone** (e.g., "80% success rate") do not reveal the specific failure modes needed to improve prompt engineering.
- **Asking users to submit feedback manually** yields low response rates and biased samples (users mostly report only severe failures).

### 2B. Anti-abuse cost enforcement

**Is this processing necessary for the stated purpose?**

Yes. Minimising it further would leave the free tier unenforceable:

- **Per-person cumulative spend** is a handful of numeric counters (micro-USD total, token counts). No alternative is feasible; billing is inherently additive.
- **Per-browser random UUID** is the minimum pinning required to correlate successive requests from the same visitor. The UUID is generated server-side from a cryptographic random source with no derivation from IP, device, or any other personal signal, so the cookie value carries no information that could be used to identify or re-identify the visitor on its own. A random per-request cookie would not enforce the cap (a visitor could refresh to bypass).
- **1-year cookie lifetime** matches the duration over which the spend counter must remain attributable. A shorter lifetime would effectively reset the cap and defeat the enforcement purpose (see §3 balancing test for justification).
- **`tocb_auth_link` (1-year, HMAC-signed)** is the minimum state needed to keep a signed-in user's cap attached to their browser across sign-out; without it a user could simply log out and start a fresh $5 counter. The cookie carries an HMAC of `{sub, exp}` with a server-side secret, is verified constant-time, and is used only for cap attribution (never as an authentication bearer).
- **Cloudflare Turnstile with a 24-hour session cookie** is a lightweight automation gate that forces re-verification at least daily, without bundling the identifier, so someone trying to sidestep the cap by clearing cookies still has to solve the widget on each reset.

**Why alternatives are insufficient:**

- **Unauthenticated rate limiting by IP** stores the raw IP (more identifying than a random UUID) and lumps every visitor behind a shared NAT/CGNAT/corporate egress into one bucket. That would let office mates lock each other out of the free tier and would require us to store a more identifying key than we do today. The per-browser UUID avoids both problems: it is not tied to the IP at all, and each browser behind the same NAT gets an independent cap.
- **Requiring sign-in to use AI features at all** would prevent first-time visitors from trying the Service; a material accessibility regression.
- **Dropping the cap entirely** exposes the shared AI-spend budget to automated abuse; one badly-behaved visitor can exhaust it.

## **3\. Balancing Test**

**Legitimate interest of the controller:**

1. *AI improvement.* Improving AI quality directly benefits all users of the tool. The Theory of Change builder's core value proposition depends on the AI assistant working well. Without real usage data, the AI cannot be meaningfully improved.
2. *Cost enforcement.* Operating a free tier is only sustainable if abuse is bounded. Per-visitor spend tracking is the minimum required to run the Service at all; dropping it would force us to either gate all AI features behind a sign-in wall or take down the Service when abuse spikes.

**Reasonable expectations of data subjects:**

- Users of an AI-powered tool **reasonably expect** that their interactions with the AI may be used to improve it. This is standard practice for AI-assisted applications.
- Users of a free AI-powered tool **reasonably expect** that the provider will meter usage to prevent abuse; it is the direct trade-off for not charging them. The per-visitor spend counter and a per-browser anonymous identifier are the minimum processing needed to deliver a "$5 free to try" experience.
- The tool's privacy policy clearly discloses both activities, including all three first-party cookies (`tocb_actor_id`, `tocb_auth_link`, `tocb_anon`) and their respective lifetimes.
- Users can opt out of AI-improvement logging at any time via their account privacy settings, with immediate effect. Opt-out is enforced both in the browser and on the server. *The anti-abuse counter itself is not opt-out-able: it is intrinsic to the free tier; users who want to avoid it can sign in or use Bring-Your-Own-Key.*

**Impact on data subjects:**

- **Low impact.** The data is identified by opaque IDs rather than names or email addresses: sessions are identified by random UUIDs, and users by Auth0 user IDs (opaque identifiers that cannot be resolved to a person without access to the Auth0 management system). No email addresses or names are stored in the logging tables.
- Chat messages contain the user's prompts to the AI, which may reference project details. This content is not stored elsewhere in the application and is the most sensitive element of the logging data.
- Graph snapshots contain the same type of data as the saved chart, but capture the temporal evolution of edits rather than just the final state.
- Cost-accounting records contain only numeric counters and an opaque identifier. No message content.
- The anonymous-visitor identifier (`tocb_actor_id`) is a random UUID generated by the server's cryptographic random source, not derived from the IP address or any other personal signal. It carries no information that could be used to identify or re-identify the visitor on its own; it is not transmitted to any third party.
- The `tocb_auth_link` cookie is a server-HMAC-signed payload containing only the Auth0 user ID and an expiry. It reveals no information an attacker could not already obtain from reading the browser's Auth0 session data, and is consulted only for cap attribution.
- This is a **small community tool** used primarily by the effective altruism community, not a large-scale commercial platform. The volume of data and number of affected individuals is small.

**Justification for the 1-year cookie lifetime:**

The `tocb_actor_id` cookie has a sole purpose: it is the key under which `user_api_usage.cost_micro_usd` accumulates for a free-tier visitor, against a $5 lifetime cap. Shortening its Max-Age would let a visitor reset the cap to zero simply by waiting for the cookie to expire (or by clearing it, which they can already do), defeating the anti-abuse purpose that justifies the processing in the first place. The cookie value is a random UUID with no personal-data derivation, stored `HttpOnly` / `Secure` / `SameSite=Lax`, and carries no information beyond "this browser has been seen before." Given that the impact on the data subject is low (no identifying content, browser-scoped, clearable at will) and the alternative is no cap at all, the 1-year lifetime is proportionate to the legitimate interest.

The `tocb_auth_link` cookie's 1-year Max-Age is justified on the same reasoning: it is a cap-attribution anchor, refreshed every time the user signs in, and revealing only the Auth0 user ID an attacker could already read from the same browser's session.

**Balancing outcome:**

The legitimate interest of the controller (improving AI quality for all users; sustainably operating a free tier) outweighs the minimal impact on data subjects, given the safeguards in place. The data collected is closely related to the service being provided, users reasonably expect it, and robust opt-out mechanisms are available. Notably, the current per-browser-UUID scheme is more privacy-preserving than an IP-based approach (no PII derivation, no NAT-peer collisions), which strengthens rather than weakens the balancing in favour of the processing.

## **4\. Safeguards**

The following safeguards are implemented to protect data subjects' rights:

1. **Opt-out mechanism (AI improvement logging):** Users can opt out of AI-improvement data collection at any time via their account privacy settings. Opting out immediately stops all logging for that user and ends any active session. The preference is enforced both client-side and server-side, and persists across devices for authenticated users. The cost-accounting counter is not opt-out-able because it is part of the basic service (free-tier gating), but users can sign in or use BYOK to exit the anonymous-visitor path entirely.

2. **Data minimization:**

   - No email addresses are stored in logging tables or cost-accounting tables.
   - Only opaque identifiers (random UUIDs, Auth0 user IDs) are used as keys: not names, not email addresses, not IP addresses.
   - User agent strings are collected only for debugging compatibility issues.
   - The anonymous-visitor identifier is a random UUID generated per browser with no derivation from IP or any other personal data; it carries no information that could be used to re-identify the visitor on its own.
   - The `tocb_auth_link` cookie contains only the Auth0 user ID plus an expiry, HMAC-signed with a server-side secret, and is consulted only for cap attribution (never substituted for a JWT).

3. **Defined retention periods:**

   - AI-improvement logging data is automatically deleted after 24 months.
   - The `tocb_actor_id` identifier cookie has a 1-year Max-Age; the `tocb_auth_link` cookie has a 1-year Max-Age (refreshed on every sign-in); the `tocb_anon` Turnstile-session cookie has a 24-hour Max-Age.
   - Per-visitor spend records for anonymous visitors are garbage-collected after the identifier cookie expires and no further activity is observed under that identifier.

4. **Deletion on request:** Users can request deletion of their data at any time. Since sessions are linked to Auth0 user IDs, data associated with an authenticated user can be identified and removed. Anonymous visitors can effectively force deletion by clearing cookies and local storage: once `tocb_actor_id`, `tocb_auth_link`, and `tocb_anon` are gone from the browser, the spend row is unreferenceable and will be garbage-collected.

5. **Access controls:** Logging and cost-accounting data are stored in the same secured database as chart data, subject to the same access controls and infrastructure security measures.

6. **Transparency:** The privacy policy clearly describes what data is collected, why, and how to opt out. The privacy popup shown to new users explains the AI-improvement data collection and offers an opt-out checkbox.

7. **No third-party sharing:** Logging and cost-accounting data are used solely internally (AI improvement, cap enforcement, security diagnostics) and are not shared with third parties.

## **5\. Outcome**

Based on this assessment, **legitimate interest under Art. 6(1)(f) GDPR** is the appropriate legal basis for both processing activities (AI-improvement logging and anti-abuse cost enforcement), subject to the safeguards documented above.

The processing is necessary for the legitimate interests of improving AI quality and sustainably operating a free tier; users reasonably expect it; the impact is minimal due to data minimization (random per-browser UUIDs with no IP derivation, Auth0 user IDs with no PII, no message content in the cost-accounting table); and a clear opt-out mechanism is provided for AI-improvement logging. The anti-abuse cookies are intrinsic to the free tier and cannot be opted out of without exiting the anonymous-visitor path (sign in or use BYOK).

This assessment should be reviewed annually or whenever the scope of data collection changes materially.
